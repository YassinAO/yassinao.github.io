API & Admin paneel

Voor het realiseren van de Simac app zijn er meerdere user stories geschreven dat dienen als de functionaliteiten voor de app om het gebruikelijk te maken voor de doelgroep. Deze user stories geven een beter beeld van wat er op de app mogelijk moet zien, voor wie en waarom.

De user stories heb ik gerealiseerd in de back-end API, zodat de gebruikers vanuit de front-end gebruik kunnen maken van deze functionaliteiten.

HINT: Je kunt kleine afbeeldingen vergroten door er op te klikken.

Aanpak

Door gebruik te maken van de geschreven user stories heb ik meerdere functionaliteiten voor de app gerealiseerd. Hieronder laat ik een aantal van deze user stories zien

  • Als gebruiker wil ik recepten per categorie inzien, zodat ik alleen de recepten waar ik om vraag krijg te zien.
  • Als gebruiker wil ik weten welke recepten zijn gemaakt per medewerker, zodat ik weet welk recepten een bepaalde collega heeft gemaakt.
  • Als gebruiker wil ik recepten kunnen toevoegen, zodat ik mijn heerlijke recepten kan delen met andere.
  • Als gebruiker wil ik recepten kunnen wijzigen, zodat ik fouten kan corrigeren.
  • Als gebruiker wil ik recepten kunnen verwijderen, zodat mijn eigen recepten niet meer voorkomen in de app.

Data seeder recepten & evenementen

Voor prototyping en testen is het belangrijk dat er data is waarmee geintracteerd kan worden. Het bouwen van data seeders helpt om snel en gemakkelijk de app te vullen met dummy data om het een levend gevoel te geven.

Door gebruik te maken van een Python module genaamd Faker heb ik dummy data weten te genereren voor recepten, evenementen en gebruikers. Hiermee wordt er tijd bespaard bij het vullen van de app met content dat vaak nodig is bij het prototypen en testen van een applicatie.

Wanneer groepsgenoten het project clonen hebben zij een lege database. Dus een data seeder helpt hen meteen om hun database te vullen met content voor de app.

Toevoegen recepten & evenementen API

Gebruikers hebben de mogelijkheid om content toe te voegen op de app. Met API mutations is het vanuit de front-end mogelijk om een create request te maken naar de API.

Deze request wordt aangeroepen zodra de gebruiker op de knop klikt dat een recept of evenement toevoegt. De data wordt meegestuurd in de request en wordt gecontroleerd op geldigheid (soort data, lengte, etc). Als de geldigheid klopt wordt de data opgeslagen. Deze data is vervolgens gekoppeld aan de gebruiker die de content heeft toegevoegd.

Wijzigen recepten & evenementen API

Gebruikers hebben de mogelijk om hun eigen content aan te passen op de app. Dit is vooral om fouten te corrigeren zoals verkeerde informatie in een recept.

Door update mutations te bouwen in de API bied ik deze mogelijkheid in de front-end van de app. Hierbij houd ik rekening met of de content gekoppeld is aan de ingelogde gebruiker om te voorkomen dat mensen elkaars content gaan aanpassen.

Net zoals het toevoegen van data wordt dit gecontroleerd op geldigheid. Als dit klopt wordt de aanpassing verwerkt in de database en vervolgens in de app zelf.

Verwijderen recepten & evenementen API

Gebruikers hebben de mogelijkheid om data te verwijderen van de app. Hierbij wordt er wel gekeken of de gebruiker toestemming heeft om dit te doen. Dit kan bepaald worden op de rol van de gebruiker (moderator, administrator) of wanneer de content is gekoppeld aan de ingelogde gebruiker. Als de gebruiker een recept heeft gemaakt kan hij/zij deze ook verwijderen, omdat het gekoppeld is.

De delete mutation is in de API verwerkt om de gebruiker deze mogelijkheid te geven in de front-end van de app. Als de validatie klopt, dus heeft juiste rol of is gekoppeld aan de content wordt dit verwerkt.

Beheren van de applicatie

De app moet up-to-date blijven en beheerd worden, zodat het geschikt blijft voor de doelgroep. Hiervoor heb ik een admin omgeving ingericht, zodat Simac de app kan beheren vanuit een admin rol.

Hierbij geef ik de mogelijkheid dat de admin alle content kan toevoegen, wijzigen, verwijderen en inzien. Ook is het mogelijkheid om gebruikers bijvoorbeeld op inactief/actief te zetten en hen meer permissie te geven op het platform indien nodig.

Conclusie

Door deze functionaliteiten te bouwen in de API bied ik de front-end de mogelijkheid om content te manipuleren en te verwerken. Verder is er voor de beheerder mogelijkheid om content bij te houden via de admin panel en API playground.

Via een usability test wil ik deze functionaliteiten testen met de doelgroep. Deze test heeft de focus op usability waarbij ik erachter wil komen of het beheren van de applicatie gebruiksvriendelijk is en waar nog ruimte ligt voor verbetering.
