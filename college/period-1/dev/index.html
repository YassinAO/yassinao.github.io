Portfolio S6

App Back-end

De Simac app is visueel uitgewerkt en er is een MOSCOW lijst gemaakt met functionaliteiten die momenteel van belang zijn. Met behulp van deze middelen heb ik gewerkt aan de back-end van de app.

Het idee is om een platform te bouwen om sociale bedrijf evenementen te organiseren. Medewerkers hebben de mogelijkheid om deel te nemen aan deze evenementen maar ook om hun eigen evenementen te organiseren. Dit is onderverdeelt in twee delen food & drink en welzijn & virtual reality. HINT: Je kunt kleine afbeeldingen vergroten door er op te klikken.

Aanpak

Door het opstellen van een MOSCOW lijst is er bepaald welke functionaliteiten in het platform komen. Verder heb ik gekeken naar de jusite development stack om deze functionaliteiten te kunnen realiseren. Hierbij is er rekeningen gehouden met kennis, ervaring, betrouwbaarheid en ondersteuning.

Om het platform op een goede manier te realiseren en geen problemen op te lopen in latere fases van de realisatie is er de juiste development stack gekozen.

Front-end

Voor de frontend wordt er gebruikt gemaakt van React Native. De reden voor React Native is de ondersteuning van meerdere platformen. Denk hierbij aan mobiel, tablet/Ipad, laptop/desktop. Er zijn bepaalde evenementen die handiger kunnen zijn voor mobiel en tablet/Ipad gebruik, denk hierbij aan het koken. Je wilt niet als gebruiker je hele desktop meenemen in de keuken. Voor andere evenementen kan een laptop/desktop handiger zijn zoals evenementen die gekoppeld zijn aan virtual reality.

React native geeft de mogelijkheid om de code maar één keer te schrijven en meteen ondersteuning te hebben voor al deze verschillende platformen.

Back-end

Voor de backend is er gekozen voor Django. Een van de redenen voor Django is door de out of the box adminpaneel die snel en gemakkelijk aangepast kan worden. Dit betekend dat het platform makkelijk beheerd en onderhoud kan worden, ook nadat het project is afgerond. Verder biedt Django goede documentatie en ondersteuning met frontend frameworks zoals React Native. Django heeft bibliotheken die gebruikt kunnen worden om de communicatie tussen Django en de frontend makkelijker te maken met bijvoorbeeld graphene of ariadne.

API

Voor de API is er gekozen voor GraphQL. Dit heeft meerdere goede redenen. GraphQL zorgt er namelijk voor dat query’s vanuit een endpoint komen. Dit betekend dat de ‘url’ waarvan gebruikt wordt altijd hetzelfde is voor elk API-call. Dit is namelijk met REST API niet het geval. Dit maakt het onderhouden van de API een stuk makkelijker. Verder zorgt GraphQL ervoor dat je geen problemen krijgt met over & underfetching. Dit betekend dat er specifiek aangegeven kan worden welke data je vanuit de API wilt halen en hiervoor ook geen meerdere calls voor hoeft te doen. Bij REST API is dit een probleem omdat je niet specifiek kunt aangeven welke data je wilt en vaak meerdere calls moet doen om alle data op te halen die nodig is. GraphQL zorgt er dus voor dat de calls een stuk sneller zijn door de precieze in het ophalen en verwerken van data.

Development & productie configuratie

Als eerste stap ben ik begonnen met de installatie, structuur en instellingen voor het project, dat vervolgens op GitHub is gezet met instructies voor andere groepsgenoten, zodat zij de backend op hun eigen machine kunnen draaien indien nodig. Bij de instellingen is er vooral onderscheid gemaakt tussen productie en development. Dit houdt in dat andere configuraties worden meegegeven wanneer het project in development is (dus het realiseren) en wanneer het in productie is (dus live en in gebruik). Dit is belangrijk omdat er met gevoelige data wordt gewerkt en test omgevingen die niet op een live platform terug moeten komen.

Commit
PostgreSQL & pgAdmin4

Via pgAdmin4 heb ik een server gemaakt waarop de database draait. Voor deze database heb ik credentials aangemaakt, zodat het alleen toegangelijk is met de juiste username en password om onbevoegde tegen te houden. Deze credentials kunnen in de development en productie configuratie meegegeven worden om toegang te krijgen tot de database.

Evenementen & recepten model

De database structuur is ontstaan door na te denken over hoe het platform momenteel in elkaar zit qua gewenste functionaliteiten uit MOSCOW. Er kan dus verschillende evenementen georganiseerd worden waar medewerkers aan kunnen deelnemen. Hierbij heb ik nagedacht welke soort informatie hierbij komt kijken, zoals naam, datum, tijd, aantal deelnemers van evenement. Dit soort informatie is dus in het database ontwerp teruggekomen.

Door gebruik te maken van Django heb ik models aangemaakt die dit soort data defineren. Met de hulp van migrations heb ik deze models omgezet tot een database structuur. Via Python shell heb ik de database getest op CRUD (create, read, update , delete) door te controleren of data wordt gemanipuleerd.

Conclusie

Door de configuratie, server en database opzet heb ik de basis structuur van de back-end voor de app gerealiseerd. Hiermee kan de volgende stap genomen worden om de verdere functionaliteiten erin te bouwen zoals de API die met de front-end zal communiceren

Met deze stap komt de front-end tot leven, doordat het data kan manipuleren doormiddel van CRUD (create, read, update, delete).
